## 🎯 Propósito

Este documento define la arquitectura de la interfaz de RemapH V3.
Describe la organización de la UI, la responsabilidad de sus componentes y las reglas que deben seguir todas las pantallas del programa.
No describe la lógica del remapeador. Esa pertenece al Runtime.
No describe la comunicación interna. Esa pertenece al Core.

# 🎯 1. Objetivos

La interfaz tiene una única misión:
Permitir al usuario crear, modificar y comprender sus remapeos de la forma más rápida y clara posible.
La UI nunca ejecuta remapeos.
La UI nunca conoce reglas del Runtime.
La UI nunca contiene lógica del sistema operativo.

---

# 🏛️ 2. Organización general

La UI se organiza mediante componentes independientes.

```
Aplicación
│
├── Toolbar
├── Tabla
│   ├── Cabecera
│   └── Filas
├── StatusBar
├── Popups
└── Ventanas
```

Cada componente posee una única responsabilidad.
La comunicación entre componentes siempre se realiza mediante el Core.

---

# 🧩 3. Estructura de la interfaz

La pantalla principal se divide en cuatro zonas.

## 🔹 Toolbar

Responsable de:

• perfiles
• estado global
• botones generales
• configuración

No conoce las filas.

---

## 🔹 Tabla

Responsable de mostrar los remapeos.
Está formada por:

• cabecera
• filas

No contiene lógica de ejecución.

---

## 🔹 StatusBar

Responsable de mostrar información contextual.

Ejemplos:

• instrucciones
• advertencias
• conflictos
• errores
• información temporal

Nunca muestra ventanas emergentes.

---

## 🔹 Popups

Responsables de ofrecer selecciones rápidas.
Son reutilizables.
Nunca contienen lógica del negocio.

---

# 🪪 4. Filas

La fila representa la unidad visual más importante del programa.
Una fila corresponde exactamente a un remapeo.
Cada fila posee:

• identidad propia
• ContextoFila
• componentes independientes

Una fila puede destruirse completamente y volver a construirse sin perder su identidad.

---

# 🧱 5. Componentes

Todo componente debe pertenecer a una de estas categorías.

## 🟢 Activos

Reciben interacción del usuario.
Ejemplos:

• Tipo
• Estado
• App
• Color
• Trigger

Su función es generar hechos.
No modifican otros componentes directamente.

---

## 🟡 Reactivos

Escuchan eventos.
Actualizan su contenido cuando corresponde.
Ejemplos:

• Acción
• Ejecución
• futuros componentes dependientes

Nunca generan cambios en otros componentes por iniciativa propia.

---

## ⚪ Neutros

Solo organizan la interfaz.
Ejemplos:

• Layout
• Tabla
• Cabecera
• Contenedores
• Separadores

---

# 🔄 6. Componentes dinámicos

Algunos componentes funcionan como contenedores.
El contenedor permanece.
Su contenido cambia.
Ejemplo:

```
Acción
┌──────────────────┐
│ Capturar tecla   │
└──────────────────┘
↓
┌──────────────────┐
│ Multimedia       │
└──────────────────┘
↓
┌──────────────────┐
│ Coordenadas      │
└──────────────────┘
```

El contenido anterior siempre se elimina completamente antes de construir el nuevo.
No se reutilizan componentes incompatibles.

---

# 👆 7. Flujo de interacción

La UI sigue siempre el mismo patrón.

```
Usuario
↓
Componente activo
↓
Core
↓
Evento
↓
Componente reactivo
↓
Actualización visual
```

La UI nunca envía órdenes directamente entre componentes.

---

# 🪟 8. Ventanas

Cada ventana constituye un módulo independiente.
Ejemplos futuros:

• Multimedia
• Coordenadas
• Macros
• Programas
• Preferencias
• Joystick

Las ventanas nunca conocen otras ventanas.
Cada una administra únicamente su propio contenido.

---

# 📢 9. Barra de estado

La barra inferior constituye el canal principal de comunicación con el usuario.
Ejemplos:

• "Seleccione una tecla."
• "Ahora haga clic en una coordenada."
• "Remapeo activado."
• "Existe un conflicto."

Debe evitarse el uso de cuadros de diálogo para mensajes informativos.

---

# ✅ 10. Buenas prácticas

Antes de crear un nuevo componente responder siempre:
¿Tiene una única responsabilidad?
¿Pertenece realmente a la UI?
¿Debe reaccionar mediante eventos?
¿Es independiente del Runtime?
¿Puede destruirse y reconstruirse sin afectar a otros componentes?
Si todas las respuestas son afirmativas, el componente probablemente pertenece a la arquitectura correcta.

La fila es la unidad principal de configuración. Las ventanas son excepciones para configuraciones complejas.
